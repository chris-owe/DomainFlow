## 一、产品定位

### 一句话定位

**按领域组织的个人工作台，任务和知识自然共生。**

混乱之中，流向清晰——但不是"一次做一件事"，而是在每个领域里有序推进任务、持续沉淀知识。

### 目标用户

个人用户，优先解决自己同时在多重角色下（工作、学习、创作）缺乏统一管理工具的痛点。

### 核心意义

用户同时扮演以下角色，信息流在这些角色之间交叉流动：

- **开发者** — 工作上的开发任务
- **项目跟进者** — 项目管理和跟进
- **学习者** — 考试、技能拓展学习
- **内容创作者** — 公众号、视频号、抖音、小红书等社媒平台的内容输出和技能沉淀

现有的工具要么只管任务（Todoist）、要么只管知识（Obsidian），没有一个工具能把这两条线在一个地方统合起来。域流要解决的就是这个问题。

## 二、核心数据模型

域流的数据模型建立在"领域"之上，领域是最高容器。

```
领域 (Domain)
  ├── 笔记 (Note)
  │     ├── 内容 (markdown)
  │     ├── 标签 (Tag)
  │     └── 关联的任务 (Task)
  │
  ├── 任务 (Task)
  │     ├── 标题 + 描述
  │     ├── 状态 (待办/进行中/已完成)
  │     ├── 截止日期
  │     ├── 优先级
  │     ├── 标签 (Tag)
  │     └── 关联的笔记 (Note)
  │
  └── 标签 (Tag) — 跨领域可复用
```

### 关键规则

1. **笔记和任务是独立对象**，但可以互相引用/关联
   - 一条笔记可以关联多个任务
   - 一个任务可以关联多条笔记
2. **每一个笔记和任务必须属于一个领域**（默认领域兜底）
3. **标签跨领域**，一个标签可以在多个领域中使用，用于快速筛选
4. **领域本身也可以嵌套**（一级即可，不建议太深）

### 对象字段定义

**笔记 (Note)**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| domain_id | UUID | 所属领域 |
| title | string | 标题 |
| content | markdown | 正文 |
| tags | Tag[] | 标签列表 |
| linked_tasks | Task[] | 关联的任务 |
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

**任务 (Task)**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| domain_id | UUID | 所属领域 |
| title | string | 标题 |
| description | string | 描述（选填） |
| status | enum | pending / in_progress / done |
| priority | enum | P0-P3 |
| due_date | datetime | 截止日期（选填） |
| is_recurring | boolean | 是否为重复任务 |
| recurring_days | int[] | 重复星期几，如 [1,3,5] 表示周一三五（0=周日, 1=周一 ... 6=周六） |
| tags | Tag[] | 标签列表 |
| linked_notes | Note[] | 关联的笔记 |
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

**重复任务规则**：若 `is_recurring = true`，则每次完成任务后，根据 `recurring_days` 自动计算下一次日期并生成新的待办实例。例如日报设为 `[1,2,3,4,5]`（周一至周五），每天完成后次日自动生成新的待办。

**领域 (Domain)**

系统首次使用时，默认创建四个领域，用户可按需增删改：

| 默认领域 | 用途 |
|---------|------|
| 工作 | 开发任务、项目管理、技术文档 |
| 考试 | 学习计划、笔记、复习 |
| 技能 | 技能拓展、教程笔记、练手项目 |
| 内容创作 | 公众号/视频号/抖音等选题、素材、排期 |

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| name | string | 名称 |
| description | string | 描述 |
| color | string | 标识色 |
| icon | string | 图标 |
| created_at | timestamp | 创建时间 |

**标签 (Tag)**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| name | string | 名称 |
| color | string | 颜色（选填） |

## 三、核心闭环

```
收集 → 关联 → 执行 → 沉淀 → 回顾
```

| 环节 | 说明 |
|------|------|
| **收集** | 快速记录——笔记、任务、灵感，1 秒入口不让念头流失 |
| **关联** | 把新产生的东西丢到对应的领域里，连接相关笔记和任务 |
| **执行** | 今日待办视图，推进任务完成 |
| **沉淀** | 任务完成后关联/补充笔记，知识自然积累 |
| **回顾** | 看领域里的任务进度和知识增长情况 |

相比旧版的区别：加入了"关联"和"沉淀"——**强调任务和笔记的网状连接**，这是域流区别于纯 Todo 工具的核心。

## 四、MVP 功能分级

### P0 — 第一版必须有

| 功能 | 说明 |
|------|------|
| **领域管理** | 创建/编辑/删除领域，配置颜色和图标 |
| **快速记任务** | 一句话创建任务，选填领域、截止日期、优先级 |
| **写笔记** | Markdown 编辑器，支持基础格式 |
| **今日待办** | 聚合所有领域中今天要做的任务 |
| **完成任务** | 点击打勾，有完成的正反馈 |
| **查看任务列表** | 按领域筛选，按状态/日期/优先级排序 |

### P1 — 第二版加

| 功能 | 说明 |
|------|------|
| **笔记-任务关联** | 笔记中引用任务，任务详情中查看关联笔记 |
| **标签系统** | 创建标签，笔记和任务均可打标签，按标签筛选 |
| **快速捕获（Inbox）** | 全局入口，快速记下任何东西（笔记或任务），暂存到待归类区。每条内容进入收集箱时自动标记 7 天倒计时，超时未归类则高亮提示 |
| **搜索** | 全局搜索笔记标题和内容、任务标题 |
| **编辑/删除笔记和任务** | CRUD 完整支持 |

### P2 — 后续迭代

| 功能 | 说明 |
|------|------|
| **周回顾** | 本周完成任务数、新增笔记数 |
| **日历视图** | 按日历查看所有带截止日期的任务 |
| **Notion 风格 Block 编辑器** | 更自由的笔记编辑体验 |
| **多端同步** | Web + App 数据互通 |
| **数据导出** | 导出笔记（Markdown）、任务（CSV） |

## 五、产品形态与技术方案

### 第一版：Web 端

快速出 Web 版，自己先用起来。

### 技术栈

| 层次 | 选型 |
|------|------|
| 框架 | React 18+ / Vite |
| 语言 | TypeScript |
| 样式 | Tailwind CSS |
| 后端 & 数据库 | Supabase（PostgreSQL） |
| 认证 | Supabase Auth（邮箱登录即可） |
| 编辑器 | React-Markdown 或 TipTap（基础 markdown） |
| 部署 | Vercel / Netlify |

### 为什么这套选型

- 一个人开发，不需要自己写后端接口，Supabase 提供数据库 + Auth + API
- TS 兜住复杂的数据模型关系
- 将来要做 App，Supabase 有客户端 SDK，Web 的 API 层可以直接复用

## 六、UI 设计方向

### 页面结构

```
┌──────────────────┬───────────────────────────────────┐
│   侧边栏           │              主内容区              │
│                  │                                   │
│  ✅ 今日待办      │  ← 全局视图，不受领域限制           │
│  📥 所有内容      │  ← 跨领域，所有笔记和任务混排      │
│  ──────────      │                                   │
│  📁 工作         │  点击某个领域后，主区域显示该领域内容 │
│  📁 考试         │  领域内分三个页签：                  │
│  📁 技能         │  [全部] [笔记] [任务]               │
│  📁 内容创作     │                                   │
│  ＋ 添加领域     │                                   │
│  ──────────      │                                   │
│  📥 收集箱       │  临时暂存，待归类                   │
├──────────────────┴───────────────────────────────────┤
│              全局快速添加栏（悬浮按钮或底部）            │
└───────────────────────────────────────────────────────┘
```

选中某个领域后，主内容区页签：
- **全部** — 该领域的笔记 + 任务混排（时间倒序）
- **笔记** — 只看笔记
- **任务** — 只看任务

不选领域时（所有内容）：
- 所有领域的笔记和任务混排（时间倒序）

### 设计语言

延续"清澈、有序"的感觉：
- 主色：冷色系，沿用清流的青色基调 #06B6D4
- 每个领域用不同颜色标识，视觉上可区分
- 极简风格，内容优先
- 今日待办视图要有"打勾"的仪式感

## 七、用户工作流（典型场景）

### 场景 1：日常任务管理

1. 早上打开域流 → 看到"今日待办"（所有领域）
2. 今天工作上有 3 个开发任务，考试有 1 个章节复习
3. 按优先级开始做，完成一个点一个勾
4. 中途接到新的工作任务 → 快速记一条任务到"工作"领域
5. 晚上看一眼今天完成了什么

### 场景 2：知识沉淀

1. 学完一个章节 → 在"考试"领域写一条笔记
2. 这条笔记关联到一个"复习"任务
3. 下次复习时打开任务就能看到笔记
4. 知识积累多了，同一个领域里可以按标签筛选

### 场景 3：内容创作管线

1. 刷到一个灵感 → 快速记到"内容创作/选题库"笔记
2. 决定做一期 → 创建任务"拍摄XXX视频"，关联选题笔记
3. 创作过程中查阅关联的素材笔记
4. 发布后把数据记录到笔记，方便复盘
5. 这条完整链条都在"内容创作"领域里完成

## 八、Roadmap

- 领域管理
